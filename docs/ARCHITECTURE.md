# Astro-Analytical Engine: Master Architecture Document

**Версия:** 5.1 (Thin Client, Multi-Stage Infra & AGPL-Safe)
**Формат разработки:** Monorepo (ИИ-ориентированный подход)

Этот документ описывает техническое устройство высокопроизводительной астрологической платформы. Архитектура спроектирована с учетом защиты интеллектуальной собственности (изоляция AGPL), минимизации серверных издержек на ранних этапах и плавной эволюции от Docker Compose до высоконагруженного Kubernetes-кластера.

## 1. Итоговый технологический стек

Стек разделен на зоны ответственности: бизнес-логика (IO), математика (CPU) и эволюционная инфраструктура.

| Слой / Компонент | Технология | Роль |
| :--- | :--- | :--- |
| **Frontend (UI)** | React + Vite + Tailwind | Пользовательский интерфейс и SPA. |
| **Визуализация** | D3.js + Cubic Splines | Отрисовка карт с локальной сплайн-интерполяцией координат. |
| **API Gateway** | Go | WebSockets, JWT, Rate Limiting, бизнес-логика. |
| **Compute Core** | Rust | Выполнение математических расчетов (Stateless). |
| **Ephemeris Service**| C / Rust | Микросервис для Swiss Ephemeris (полная изоляция AGPL). |
| **Inter-service** | gRPC over TCP | Бинарные контракты внутри виртуальной сети. |
| **Message Broker** | NATS JetStream | Единый легковесный брокер очередей со Stage 1. |
| **Базы данных** | PostgreSQL + Redis + SQLite | Postgres: профили. Redis: кэш/квоты. SQLite: локальный оффлайн-кэш на клиенте. |
| **Desktop / Mobile**| Tauri / Capacitor | Тонкие клиенты без локального ядра вычислений. |

## 2. Архитектура системы и Data Flow

Система работает в режиме «Тонкого клиента» с предиктивным кэшированием. Вся тяжелая математика инкапсулирована на бэкенде.

### 2.1 Синхронный контур и Предиктивное кэширование (Динамическая Натальная Карта)
1. **Frontend** (при наличии сети) запрашивает «Эфемеридное Окно» — массив опорных точек на широкий временной диапазон (например, ±30 дней), передавая JWT.
2. **Go API** валидирует запрос и проверяет лимиты в Redis.
3. **Go API** вызывает **Rust Core** через gRPC over TCP.
4. **Rust Core** вычисляет координаты планет (например, на 00:00 и 12:00 каждых суток) через **Ephemeris Service** и отдает сжатый бинарный дамп.
5. **Frontend** сохраняет дамп в локальную БД (SQLite через плагины Tauri/Capacitor).
6. При прокрутке времени пользователем, клиент читает данные из локальной БД и использует **Естественные кубические сплайны (Natural Cubic Splines)** для математически точного сглаживания траекторий. Это обеспечивает 60 FPS, точность в периоды стационарности и оффлайн-доступ.

### 2.2 Асинхронный контур (Inverse Search)
1. Пользователь инициирует поиск сложных аспектов. **Go API** создает Job.
2. Задача пушится в **NATS JetStream**.
3. **Rust Workers** забирают задачу, проводят первичное сканирование с шагом, учитывающим максимальные угловые скорости объектов, затем применяют иерархическое сужение (метод Брента) внутри найденных интервалов и стримят результаты батчами.
4. Результаты доставляются на фронтенд через WebSocket для мгновенного отображения прогресса. В оффлайн-режиме функция блокируется (Graceful Degradation).

## 3. Стратегия защиты и изоляции

* **Юридическая чистота (AGPL):** Весь код, взаимодействующий со Swiss Ephemeris, вынесен в бинарно изолированный сервис `ephemeris-svc`. Общение идет через gRPC. Это позволяет держать основное ядро и фронтенд закрытыми.
* **Отказ от WASM на клиенте:** Решение принято для исключения утечки проприетарных алгоритмов интерпретации и устранения необходимости лицензирования фронтенда под AGPL.
* **Защита алгоритмов:** Все интерпретации (бизнес-смысл градусов) хранятся и вычисляются только на стороне Go/Rust бэкенда. На клиенте выполняется только открытая математика интерполяции.

## 4. Эволюция инфраструктуры (Deployment Roadmap)

### Stage 1: Docker Compose (MVP)
* **Цель:** Запуск на VPS (2-4 ГБ RAM) с минимальным потреблением ресурсов.
* **Связь:** gRPC over TCP между всеми компонентами на одной ноде.
* **Брокер:** NATS JetStream (один бинарник, потребление ~20 МБ RAM).
* **Трафик:** Traefik в контейнере.

### Stage 2: K3s (Scaling)
* **Цель:** Отказоустойчивость и Zero-Downtime обновления.
* **Хранение:** StatefulSets с локальным хранилищем (local-path-provisioner).
* **Брокер:** Кластеризация NATS JetStream внутри кластера.
* **Масштабирование:** Горизонтальное масштабирование Rust-воркеров.

### Stage 3: Managed K8s (High Load)
* **Цель:** Глобальная доступность и Managed-сервисы.
* **БД:** Миграция на облачные БД (Managed PostgreSQL/Redis).
* **Связь:** gRPC over TCP (Service Mesh).
* **Хранение:** S3-совместимые хранилища для архивов эфемерид.
